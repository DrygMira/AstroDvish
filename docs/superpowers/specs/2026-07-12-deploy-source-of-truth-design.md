# Дизайн: воспроизводимый деплой с фиксацией источника (deploy source-of-truth)

Дата: 2026-07-12
Ветка: `codex/shared-birth-context-ui`
Статус: дизайн на согласовании

## 1. Цель простыми словами

Сделать так, чтобы про боевой сервер всегда было понятно **что именно на нём сейчас работает** и **совпадает ли это с нашим кодом**. Выкладка новой версии — одной командой, которая сама делает бэкап, копирует код, оставляет «бирку» с версией и проверяет, что всё живо.

## 2. Проблема (что чиним)

- Сейчас деплой = ручное копирование файлов на сервер. Нигде не записано, какая версия выкачена.
- Из-за этого локальный репозиторий и прод молча разъезжаются. Реальный случай `2026-07-12`: на сервере лежал деплой от `2026-07-05`, которого не было ни в git, ни в `PROJECT_STATE.md`; пришлось сверять файлы вручную.
- Нет быстрого способа ответить «прод == мой код?» и нет единого управляемого отката.

## 3. Решение: один инструмент `scripts/deploy.py`

Python-скрипт (paramiko по SSH), запускается из корня проекта на рабочей Windows-машине. Четыре режима:

| Команда | Что делает |
|---|---|
| `python scripts/deploy.py --plan` | Сухой прогон: показать какой коммит/файлы поедут и какой бэкап снимется. Ничего не меняет. |
| `python scripts/deploy.py` | Полная выкладка (см. §4). |
| `python scripts/deploy.py --status` | В одну строку: live-версия vs локальный HEAD vs remote `dryg`. |
| `python scripts/deploy.py --rollback <backup.tgz>` | Восстановить сервер из указанного бэкапа + пересобрать. |

Целевой сервер и путь берутся из констант/`.env` инструмента: `root@45.133.18.90`, `/opt/astrodvish`, ключ `~/.ssh/auron_deploy`.

## 4. Что делает полная выкладка (по шагам)

1. **Сборка артефакта из текущего рабочего дерева.**
   Список файлов = `git ls-files` (трекнутые, текущее содержимое — то есть незакоммиченные правки тоже попадают) + `git ls-files --others --exclude-standard` (новые файлы, ещё не в git, но не игнорируемые). Пакуется в `.tar.gz`.
   - Мусор и секреты отсекаются автоматически через `.gitignore`: `.venv`, `tests/_tmp`, `*.pem`, `deploy/`, `ephe`, `.env`, `secrets.txt`.
   - Важное следствие: `.env` не попадает в артефакт → **боевой `.env` на сервере не затирается**.
2. **Гибкий guard (не блокирует, но предупреждает).**
   Если дерево грязное или коммит не запушен в `dryg` — печатается заметное предупреждение, но выкладка продолжается. Свобода деплоить что угодно сохраняется; тишины нет.
3. **Бэкап текущего live.**
   `tar czf /opt/astrodvish_backups/predeploy_<ts>_<shorthash>[_dirty].tgz` папки `/opt/astrodvish` (без `ephe`, без каталога бэкапов). Соответствует уже принятому на сервере соглашению `predeploy_*.tgz`.
4. **Фиксация версии («бирка») — до распаковки.**
   На сервер пишется `/opt/astrodvish/DEPLOYED.json`:
   ```json
   {"commit": "<full sha>", "short": "<7>", "branch": "<name>",
    "dirty": true|false, "pushed_to_dryg": true|false,
    "deployed_at": "<iso utc>", "artifact_sha256": "<sha>"}
   ```
   Если дерево грязное — рядом кладётся `DEPLOYED_uncommitted.patch` (`git diff HEAD`), чтобы даже грязный деплой был полностью восстановим.
5. **Распаковка + подчистка хвостов.**
   Артефакт распаковывается в `/opt/astrodvish` (перезапись и добавление файлов проекта). Устаревшие хвосты в корне от прежних ручных копирований (`main.py`, `api.js` не на своём месте) удаляются.
   - Поведение удаления: инструмент **перезаписывает и добавляет** файлы из артефакта и удаляет только **явно заданный список хвостов**. Произвольные «лишние» файлы на сервере не удаляются (безопасно, «не трогать лишнего»). Ограничение: если в git удалить исходный файл, на сервере он останется — такие точечные удаления делаются отдельно и осознанно. Полный бэкап (шаг 3) страхует в любом случае.
6. **Пересборка и health-gate.**
   `docker compose up -d --build`. Ждём, пока оба контейнера `healthy` и `/health` = `200` на api (`:8013`) и web (`:8014`/`:80`), с таймаутом.
7. **Провал health → авто-откат.**
   Если health не поднялся за таймаут — автоматически восстановить из бэкапа шага 3, пересобрать, вернуть внятную ошибку «деплой откатан, прод как был».

## 5. Что осознанно НЕ входит (YAGNI)

- Staging-окружение, CI/CD, worker-очередь под OOM, git на сервере, ротация секретов — это отдельные пункты плана улучшений, сюда не подмешиваются.

## 6. Тестирование

- **Юнит-тест** на чистую логику сборки: корректность списка файлов (трекнутые + новые не-игнорируемые, без мусора) и детект «грязного» дерева. Быстрый, без сети.
- **End-to-end (безопасный)**: первый реальный прогон — деплой текущего коммита, который уже идентичен серверу. Это no-op по поведению прода, но проходит весь путь бэкап→stamp→распаковка→пересборка→health→откат-при-провале и доказывает, что инструмент работает.

## 7. Критерии готовности

- `--status` корректно показывает совпадение/расхождение live vs git.
- Успешный деплой оставляет валидный `DEPLOYED.json`; грязный — ещё и `.patch`.
- Симуляция провала health приводит к авто-откату (проверяется на безопасном сценарии).
- Боевой `.env` и `ephe` на сервере не затронуты.
- Юнит-тест сборки артефакта проходит; полный `pytest` остаётся зелёным.

## 8. Технические детали

- Драйвер: Python + `paramiko` (SSH/SFTP), уже проверен в этой сессии с Windows-машины. Bash-`deploy.sh` не выбран из-за нестабильности запуска с Windows.
- Идемпотентность: повторный деплой того же дерева даёт тот же результат (артефакт по содержимому детерминирован).
- Секреты: ключ деплоя не хранится в репозитории (`*.pem`, `deploy/`, `~/.ssh/*` в `.gitignore`); инструмент читает путь к ключу из окружения/константы.
